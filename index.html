Why modules?

Present compilation model

  • Inherited from C
  • Uses preprocessor for code reuse
  • Preprocessor doesn't know language semantics

Issue: leaking names

Some code from an stdlib implementation:


      template <class _Tp, class _Allocator>
      inline _LIBCPP_INLINE_VISIBILITY void
      vector<_Tp, _Allocator>::push_back(const_reference __x)
      {
          if (this->__end_ != this->__end_cap())
          {
              __RAII_IncreaseAnnotator __annotator(*this);
              __alloc_traits::construct(this->__alloc(),
                                        _VSTD::__to_raw_pointer(this->__end_), __x);
              __annotator.__done();
              ++this->__end_;
          }
          else
              __push_back_slow_path(__x);
      }

Why it's so ugly?


        #define annotator 1
        class RAII_IncreaseAnnotator{};
        #include <vector>

Issue: leaking names

  • All names from previous headers are in scope
    • Macros
    • Classes
    • Global variables
  • Especially bad for library writers
  • Stdlib implementers can afford reserved names; others don't have this luxury

Issue: ODR violations

One Definition Rule:

A variable, function, class type, enumeration type, or template shall not be defined where a prior definition is necessarily reachable; no diagnostic is required if the prior declaration is in another translation unit.

(shall means the program is ill-formed)

Issue: ODR violations


        // a.cpp
        #include "header.h"
      

        // b.cpp
        #define _NDEBUG
        #include "header.h"
  • This could affect member layout, object size, type properties, ...
  • Easy to do accidentally
